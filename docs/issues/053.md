# 053 — Accessibility debt: Dynamic Type untested, Reduce Motion ignored

**Area:** accessibility · **Priority:** medium · **Effort:** M · **Status:** open

## Problem
Two system accessibility settings the app currently ignores:

1. **Dynamic Type.** [DESIGN.md](../DESIGN.md) promises "don't disable font scaling; layouts must tolerate larger text", but nothing in the codebase touches `allowFontScaling`/`maxFontSizeMultiplier` and no layout has been checked at accessibility sizes. Likely casualties at AX2+: fixed-height controls (`h-12` buttons, the 44pt composer buttons, `h-16` attachment thumbs), the hero stat tiles, week-glance bars, the chat bubbles' `max-w-[88%]`, and every `numberOfLines={1}` title. An athlete demographic skews older — large-text support is not a corner case for a coaching app.
2. **Reduce Motion.** All entrance/press animations (`FadeInDown` staggers on Today, spring-scale `AnimatedPressable`, `LayoutAnimation` in forms) run unconditionally. Reanimated supports `ReduceMotion.System` per animation; `LayoutAnimation` calls need an `AccessibilityInfo.isReduceMotionEnabled` guard.

## Proposal
One audit pass: run key screens (Today, Log, Coach, detail pages) at the largest accessibility text size, fix clipping with `maxFontSizeMultiplier` (cap ~1.5–2 on dense meta lines, let body text grow) and min-height-instead-of-fixed-height controls. Add `ReduceMotion.System` to reanimated configs and a reduce-motion guard around `LayoutAnimation`/stagger delays. Document both rules in DESIGN.md with the chosen caps.
