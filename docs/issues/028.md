# 028 — Date-only strings parsed as UTC shift planned workouts/events by one day in western timezones

**Area:** dates · **Priority:** high · **Effort:** S · **Status:** done

## Problem
`localDateKey` ([weekGlance.ts:37](../../src/features/today/weekGlance.ts)) does `new Date(input)` on strings and then reads **local** getters. For a date-only string like `"2026-07-20"`, JS parses it as **UTC midnight**, so in any UTC-negative timezone (all of the Americas) the local getters return **July 19**.

Every consumer of that helper inherits the off-by-one:

- `computeWeekGlance` — done/planned load lands in the wrong weekday bucket.
- `groupUpcomingByDay` ([groupUpcoming.ts](../../src/features/activity/groupUpcoming.ts)) — a workout planned for today shows under the wrong day header (or a "Today" session disappears into yesterday).
- `daysUntilLocal` ([mapEvents.ts](../../src/features/events/mapEvents.ts)) — race countdown is one day short, and an event **on** today can be dropped as "past" (`days < 0`).
- `formatActivityDate` ([mapActivity.ts:297](../../src/features/activity/mapActivity.ts)) has the same `new Date(string)` pattern for list row labels.

Whether this fires depends on whether coach-wattz sends date-only strings or full ISO timestamps for `plannedWorkout.date` / event `date` — planned-workout and calendar-event dates are classically date-only. Repro: set device timezone to `America/Los_Angeles`, plan a workout for today.

## Proposal
In `localDateKey` (and `formatActivityDate`), detect a `^\d{4}-\d{2}-\d{2}$` input and construct the date with `new Date(y, m - 1, d)` (local) instead of the string parser; keep the current path for full timestamps. Add vitest cases with a mocked TZ (`TZ=America/Los_Angeles vitest`).
