# 035 — Raw JSON parse errors leak to the athlete from generate endpoints

**Area:** cross-cutting · **Priority:** low · **Effort:** XS · **Status:** done

## Problem
[dailyCheckinApi.ts:49](../../src/features/log/dailyCheckinApi.ts) and [today/api.ts:64](../../src/features/today/api.ts) share this pattern:

```ts
} catch (e: any) {
  if (e.message) throw e;
}
```

The catch is meant to re-throw the intentional 429 quota error, but it also re-throws the `SyntaxError` from `response.json()` when the error body isn't JSON (proxy HTML 502, plain-text error). The athlete then sees `Unexpected token '<' …` instead of the friendly "Generation failed (502)" fallback — exactly what issue 001 was meant to eliminate.

## Proposal
Throw the quota error outside the try/catch (parse first, decide after), or mark it (`e.isQuota = true`) and only re-throw that; let parse failures fall through to the status-based message.
