# 054 — i18n scaffold is dead code; mixed-language experience is unowned

**Area:** i18n · **Priority:** medium · **Effort:** M (decision + sweep) · **Status:** open

## Problem
[src/i18n/](../../src/i18n) exists (`t()`, `messages/en.ts`) but **nothing imports it** — every screen hardcodes English strings. Meanwhile the server speaks the athlete's language: this session's live data showed Hungarian coach chat, Hungarian AI analysis, Hungarian step names ("Bemelegítés") inside English chrome ("STRUCTURE", "Coach cues", "AI analysis"). The result is a permanently bilingual UI for any non-English athlete, and the half-built scaffold gives false confidence that localization is "started".

## Proposal
Make a deliberate call and encode it:

- **If v1 is English-chrome-only** (reasonable): delete the unused i18n module (or park it in a branch), state the decision in [open-questions.md](../open-questions.md)/DESIGN.md, and make sure chrome copy never mixes into server content (e.g. don't prefix server strings with English labels mid-sentence).
- **If localization is wanted** (the server clearly personalizes language): route new/edited strings through `t()` from now on, backfill screen by screen, and pick the locale from the profile/instance rather than device-only.

Either way, stop the middle state where the scaffold exists but the convention is "hardcode it".
