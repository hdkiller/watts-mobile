# 042 — Planned detail: nonsense step meta ("2 bpm"), all-gray silhouette, jargon status

**Area:** detail · **Priority:** medium · **Effort:** M · **Status:** done

## Problem
Verified live on a planned HR-based run ("Treadmill Warmup Run"):

1. The "Active" step renders "20 min · **2 bpm**" — the intensity label maps a target (likely zone 2 or an HR range) into a nonsense unit. Athletes reading "2 bpm" will assume the app is broken. (Mapper: [mapActivity.ts](../../src/features/activity/mapActivity.ts) step intensity/label path.)
2. The structure-profile silhouette (issue 008/009's centerpiece) renders **all-gray equal-height bars** when step intensity doesn't parse — here every step failed to resolve, so the marquee visual adds nothing. HR-zone-based steps should resolve via the zone list that's right below on the same screen.
3. Status line "Pending · Synced" is server jargon; unclear what's pending for an athlete.

## Root cause (confirmed against coach-wattz)
Step targets carry a `units` field the mobile mapper ignores: `heartRate: { value: 2, units: '…zone…' }` means **Zone 2**, and the server resolves it to a bpm range via the athlete's `hrZones` (see coach-wattz `server/utils/workout-analysis-facts.ts` `targetUnits?.includes('zone')` branches; the web `WorkoutStepRow.vue` computes zone name + concrete target per step). `intensityFromStep` ([mapActivity.ts:558](../../src/features/activity/mapActivity.ts)) prints `${value} bpm` unconditionally → "2 bpm".

## Proposal
Read `units` in `intensityFromStep`: when it denotes a zone, emit "Z<n>" and resolve the concrete range from the payload's own `zoneSummary.bands` ("Z2 · 141–150 bpm"). Feed the zone index into `stepIntensity` so the silhouette gets zone colors for HR/pace-zone workouts, not just %FTP. Replace the raw status words with athlete copy ("Awaiting sync to device" / omit when synced). Add a mapper test with the HR-zone-structure fixture from this workout. Presentation follow-ups live in 047 (planned page) and 048 (activity page).
