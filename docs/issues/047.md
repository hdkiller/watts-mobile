# 047 — Planned detail: zones table dominates; steps should carry the zone info instead

**Area:** detail · **Priority:** medium · **Effort:** M · **Status:** done

## Problem
On [planned/[id].tsx](<../../app/(app)/planned/[id].tsx>) the "Zones · Heart rate" section renders **all 7 bands fully expanded, always** — roughly a whole viewport of reference data below the structure. Verified live: on an HR-based run the zone table is the biggest section on the page while the steps above it say "20 min · 2 bpm".

The web app (coach-wattz `app/pages/workouts/planned/[id]/index.vue` + `WorkoutStepRow.vue`) has **no standing zones table at all**: each step row carries a colored zone dot, the zone name, and the concrete target (computed from sport settings). Zones live where the athlete's eye already is — on the step. Mobile inverted this: weak step labels + a big static reference table the athlete must mentally join against the steps.

## Proposal
1. **Move zone info onto the steps** (pairs with 042's `units` fix): each step row gets its zone color bar (already there), zone name, and resolved range — "Z2 · 141–150 bpm" — using the `zoneSummary.bands` that arrive on the same payload.
2. **Collapse the zones table** once steps are self-sufficient: render it as a collapsed disclosure row ("Zones · Heart rate ▸", tap to expand), or replace it with a single compact color-ramp bar with the range labels available on tap. Field rule from DESIGN.md applies: one decision per viewport — the execution plan is the decision, the zone chart is an appendix.
3. Keep the expanded table for the no-structure case (zones are then the only content worth showing).
