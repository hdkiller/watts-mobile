# 062 — Reanimated misuse warnings: layout animations fight AnimatedPressable's own props

**Area:** animations · **Priority:** medium · **Effort:** S · **Status:** done

## Problem
Metro logs on a clean launch showed two repeated Reanimated warnings:

1. **`Properties "opacity, transform" of AnimatedComponent(Pressable) may be overwritten by a layout animation. Please wrap your component with an animated view and apply the layout animation on the wrapper.`** — exactly 10×, matching the activity list's staggered rows: [activity/index.tsx](<../../app/(app)/activity/index.tsx>) passes `entering={FadeInDown…}` straight onto [AnimatedPressable](../../src/components/AnimatedPressable.tsx), whose `useAnimatedStyle` animates the very same `opacity`/`transform` for press feedback — two animation systems racing on one node (the conflict class behind 058-style stuck transforms).
2. **`Writing to 'value' during component render`** — several per mount, source not pinpointed (first-party writes are all in effects/handlers).

Also seen: `Sending "websocketClosed" with no listeners registered` — benign (coach WS closes after listener teardown).

## Resolution (2026-07-20)
[AnimatedPressable.tsx](../../src/components/AnimatedPressable.tsx) now hoists `entering`/`exiting`/`layout` onto a wrapper `Animated.View`, keeping press opacity/transform on the inner pressable (Reanimated's recommended pattern). Verified via the runtime log registry: activity list mounts with **zero** layout-animation warnings (was 10×).

**Residual:** one "Writing to value during component render" per launch remains unattributed — chase with `configureReanimatedLogger` stack traces when convenient. `websocketClosed` stays as-is (benign).
