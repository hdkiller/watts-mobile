# 060 — Coach room switcher can strand an opaque empty sheet over the app

**Area:** Coach · **Priority:** high · **Effort:** S · **Status:** done

## Problem
Reproduced live: Coach tab → header room switcher → tap an older room. The chat loads correctly behind it, but the pageSheet **reappears as a completely blank, opaque card covering the whole screen** — no content, no buttons, and swipe-down does not dismiss it. The accessibility tree showed the healthy chat screen *underneath* (messages, composer, tab bar) while the top-most sheet exposed zero elements. Only killing the app recovered. A hard-stuck state on the daily-loop surface.

Mechanics: [RoomListSheet.tsx](../../src/features/coach/RoomListSheet.tsx) is an RN `Modal` with `presentationStyle="pageSheet"`, and `selectRoom` in [useCoachChat.ts](../../src/features/coach/useCoachChat.ts) ran `await applyActiveRoom(room)` (async room switch + message load, triggering re-renders) **before** `setRoomListOpen(false)`. A `visible` flip racing the pageSheet's own transition while the owner re-renders is a known RN iOS Modal failure mode — the UIKit sheet gets orphaned with stale (empty) content.

## Resolution (2026-07-20)
`selectRoom` and `createRoom` in `useCoachChat.ts` now call `setRoomListOpen(false)` synchronously **before any `await`**, so the Modal's `visible` flip can't race the async room switch. Verified on device: open switcher → select history room → chat renders, sheet dismisses cleanly and stays gone. If the orphaned-sheet race ever resurfaces, next step is replacing the RN `Modal` with the router's native form-sheet route (available post-049).
