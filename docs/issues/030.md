# 030 — Switching instance URL keeps old tokens and query cache

**Area:** auth/instance · **Priority:** medium · **Effort:** S · **Status:** done

## Problem
`saveInstance` in [AuthContext.tsx](../../src/auth/AuthContext.tsx) validates reachability, stores the new URL, and sets `needs_login` — but it neither clears tokens nor the TanStack Query cache (memory + AsyncStorage persister):

- The previous instance's access/refresh tokens stay in SecureStore. Until the next sign-in completes, any code path that fires a request (or the refresh path) sends the **old instance's Bearer token to the new instance's host**.
- Persisted Today/planned caches from the old instance survive and can render as the "current" data after signing in to the new instance (self-hosted → hosted switch shows another server's training data until refetch).

Self-hosted instances are first-class per AGENTS.md, so this is a real flow, and cross-host token leakage is a security smell even when the receiving server just rejects it.

## Proposal
In `saveInstance`, when the normalized URL differs from the current one: `await clearTokens()`, `queryClient.clear()`, and clear the persisted cache (persister `removeClient()` / remove the `watts.reactQuery.v1` AsyncStorage key) before setting `needs_login`.
