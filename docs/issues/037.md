# 037 — Coach chat keyboard: input bar hidden behind keyboard, no dismiss, send unreachable

**Area:** Coach · **Priority:** high · **Effort:** S · **Status:** done

## Problem
[CoachChat.tsx](../../src/features/coach/CoachChat.tsx) has no keyboard handling at all, and it's the heaviest typing surface in the app. On iOS (which never auto-resizes the window):

- Focusing the input opens the keyboard **over** the composer row — the athlete can't see what they're typing, and the send/attach buttons sit physically under the keyboard, so the message can't be sent.
- There is no way out either: the message `FlatList` sets no `keyboardDismissMode`, and there's no tap-to-dismiss, so scrolling or tapping the conversation doesn't lower the keyboard. The only escape is the swipe-down-from-input OS gesture most users don't know.
- When the keyboard opens, the list doesn't re-scroll, so the newest messages are also covered.

Every other input screen (Log, daily check-in, recovery event, settings forms, instance) already handles this via `useKeyboardOverlap` or `KeyboardAvoidingView` — chat is the one exception. Android mostly survives on the default `resize` mode (nothing is configured in app.json), so this bites iOS primarily.

Reported from device use: "when the keyboard opens I don't see the input field, and I have trouble touching outside the keyboard to send the message."

## Proposal
Standard chat-screen keyboard UX:

1. Wrap the chat column (or pad the composer) with [useKeyboardOverlap](../../src/hooks/useKeyboardOverlap.ts) — same pattern as Log — so the composer rides above the keyboard. Note the coach tab's `SafeAreaView edges={{ bottom: true }}` ([coach.tsx](<../../app/(app)/(tabs)/coach.tsx>)): subtract the bottom inset from the overlap or move the inset inside, or the bar ends up double-padded.
2. `keyboardDismissMode="interactive"` on the message `FlatList` (drag-to-dismiss like iMessage) and `keyboardShouldPersistTaps="handled"` so taps on bubbles/approval buttons work with the keyboard up.
3. On keyboard open (`keyboardWillChangeFrame`), `scrollToEnd` so the latest messages stay visible above the composer.
4. Consider `react-native-keyboard-controller` (`KeyboardStickyView`) if the interactive-dismiss + sticky composer combo fights the hook; it's the current community standard for chat composers.

## Resolution
Shipped in `CoachChat.tsx` (2026-07-20): root view uses `useKeyboardOverlap` with `paddingBottom: overlap` (measured against the container itself, so the SafeAreaView bottom inset doesn't double-pad); message `FlatList` got `keyboardDismissMode` (`interactive` on iOS, `on-drag` on Android) + `keyboardShouldPersistTaps="handled"`; a new effect scrolls to the latest message when the keyboard opens; the empty starter-prompt state became a `ScrollView` with `keyboardDismissMode="on-drag"` so dismissal also works before the first message. Verified on the iPhone 17 Pro simulator (composer above keyboard, send reachable, scroll with keyboard up, drag-into-keyboard dismisses). `react-native-keyboard-controller` not needed for now.
