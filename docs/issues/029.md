# 029 — saveTokens leaves stale refresh token / expiry behind

**Area:** auth · **Priority:** medium · **Effort:** XS · **Status:** done

## Problem
[tokenStorage.ts](../../src/auth/tokenStorage.ts) `saveTokens` only **writes** keys when values are present:

- `refreshToken: null` (login response without a `refresh_token`, or the e2e seed) keeps whatever refresh token was stored before. A later 401 then "refreshes" with a token that doesn't belong to the current session/grant — best case `invalid_grant` and sign-out, worst case resurrecting an older session.
- `expiresIn` null/undefined keeps the previous `cw.accessExpiresAt`, so the stored expiry describes a different token.

## Proposal
In `saveTokens`, treat explicit `null` as "delete": `deleteItemAsync(REFRESH_KEY)` when `refreshToken === null`, and `deleteItemAsync(EXPIRES_KEY)` when no `expiresIn`. (Keep the current "preserve on `undefined`" behavior for the rotation path in `refreshAccessToken`, which intentionally re-passes the old token.)
