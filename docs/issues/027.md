# 027 — Token refresh race between apiFetch and coachChatFetch; failed chat refresh never updates auth state

**Area:** auth/api · **Priority:** high · **Effort:** S · **Status:** open

## Problem
Two independent refresh paths exist:

- [client.ts](../../src/api/client.ts) `apiFetch` uses `singleFlightRefresh` (module-level promise) — good.
- [coachFetch.ts](../../src/features/coach/coachFetch.ts) `coachChatFetch` calls `refreshAccessToken` **directly**, outside the single-flight.

When an access token expires while the Coach tab is open, a chat request and any regular API request 401 at the same time and fire **two parallel refresh calls with the same refresh token**. If coach-wattz rotates refresh tokens (one-time use), the loser gets `invalid_grant`, its catch runs `clearTokens()`, and the athlete is silently signed out mid-session.

Two related gaps in the same paths:

1. `coachChatFetch`'s failed-refresh catch clears tokens but never calls the `onAuthFailure` handler, so `AuthContext` stays `authenticated` with no tokens — every subsequent request 401s until app restart.
2. In `apiFetch`, when the refresh succeeds but the **retried** request still returns 401 (access revoked server-side), the 401 is returned to the caller with no auth-failure transition either.

## Proposal
Export the single-flight refresh from `client.ts` (or a shared `auth/refresh.ts`) and have `coachChatFetch` use it. On any terminal refresh failure or post-refresh 401, funnel through the same `clearTokens()` + `onAuthFailure()` path so the UI lands on the login screen instead of a dead session.
